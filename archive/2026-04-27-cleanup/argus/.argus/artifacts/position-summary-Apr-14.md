# All Round 1 Positions

## Pragmatist
**Claim:** We are running a massive optimization engine against the wrong metric and carrying thousands of lines of dead weight and unnecessary validation ceremony that need to be deleted.
**Evidence:** The engine still optimizes for dollar return (`vs_bah`) and penalizes trade frequency instead of the North Star's "share-count multiplier" goal. `scripts/strategies.py` is a 3,500-line graveyard of failed strategies that `evolve.py` actively skips. `scripts/validation/` has 9 layers of complex statistical checks that likely catch nothing a basic holdout test wouldn't. `strategy_engine.py` and `backtest_engine.py` are two files doing the job of one.
**Biggest opportunity:** Mass deletion. Delete the dead strategies in `scripts/strategies.py`. Delete the 9-layer validation ceremony in favor of a simple 2-layer out-of-sample test. Merge the two engines into a single entry point.
**Biggest blocker:** The 3,300-line brittle string formatting in `scripts/pine_generator.py` is a massive translation tax, and the optimization engine is currently validating against deprecated rules.

## Futurist
**Claim:** The string-template architecture in `pine_generator.py` is an architectural dead end that will completely block the North Star goal of true automated strategy discovery within 18 months.
**Evidence:** Trace this forward: the codebase's North Star explicitly demands "automated strategy discovery," but currently, the Python backtester and the Pine Script generator maintain parallel, hardcoded logic. Because there is no shared Abstract Syntax Tree (AST) or Domain Specific Language (DSL) defining the indicator topology, the evolutionary engine is structurally limited to just tuning parameters for human-written templates. When the engine inevitably needs to discover novel logic—like combining an RSI entry with a TEMA exit—a human will have to manually write both the Python and Pine representations. This dual-maintained string-formatting architecture means the system cannot autonomously evolve new strategies.
**Biggest opportunity:** Introducing a shared configuration object or DSL that represents strategy logic abstractly. If both the Python backtester and Pine generator compile from the same fundamental source of truth, the system can autonomously mutate indicator topologies, shifting from parameter optimization to genuine programmatic logic discovery.
**Biggest blocker:** The engine's evolutionary state is locked into the raw file bytes (`_ENGINE_HASH` in `evolve.py`) and stored in a single, non-concurrent `hash-index.json` file. Any structural refactoring toward a shared AST will instantly invalidate thousands of hours of historical compute cache, and the single-file JSON index guarantees the team cannot scale compute horizontally to rebuild that history or explore the expanded search space.

## Craftsman
**Claim:** The system's foundational abstractions are lying about what they are, most critically in the inversion of `backtest_engine.py` (which is actually a specific strategy) and `strategy_engine.py` (which is the actual engine), creating a fractured domain model that guarantees any engineer will build a false mental model of the architecture.
**Evidence:** The core domain entities `Trade` and `BacktestResult` are defined twice as independent `@dataclass` objects with different schemas across the two "engine" files, and both files implement their own redundant indicator math (EMA, RMA, SMA). Furthermore, the codebase vocabulary (`vs_bah_multiple`) contradicts the team's actual mental model and charter ("share-count multiplier"), forcing reliance on explanatory comments instead of honest naming.
**Biggest opportunity:** Unifying the domain model by resolving the engine naming inversion and establishing a single source of truth for core entities and indicator math, which will instantly align the code's architecture with its actual behavior and the North Star.
**Biggest blocker:** The cognitive debt embedded in the current misleading names and duplicated logic; engineers are forced to constantly translate between the charter's goals (share accumulation) and the code's obsolete vocabulary (`vs_bah`), making every modification slower and more prone to conceptual errors.

## Accelerator
**Claim:** A new engineer cannot safely ship a change to the core engine today because the absence of a fast, localized unit test suite makes fearless refactoring impossible.
**Evidence:** The project relies entirely on a heavy, data-science-style validation pipeline (`scripts/validation/`) to test strategies, but lacks a basic `pytest` suite for the Python backtester itself. This means engineers have no immediate, low-latency feedback loop to verify they haven't broken basic execution logic when modifying files like `backtest_engine.py`.
**Biggest opportunity:** Decoupling code validation from strategy validation by introducing a sub-2-second unit test suite that stubs out the data layer, giving engineers the fast feedback loop required to safely evolve the system.
**Biggest blocker:** The completely manual deployment pipeline, which requires human copy-pasting of generated code into the TradingView UI. This acts as a hard ceiling on velocity, introduces final-mile error risk, and fundamentally prevents rapid, automated rollbacks.

## Exploiter
**Claim:** The validation pipeline's trust model is fundamentally inverted, blindly trusting external data streams and local state files without integrity checks, allowing an attacker to silently curve-fit the generated trading strategies to a poisoned reality without ever needing arbitrary code execution.
**Evidence:** The system pulls data from Yahoo Finance and FRED directly into the backtest engine without bounds checking or anomaly detection, while simultaneously relying on unverified local state files like `TECL-markers.csv` and `leaderboard.json` to determine strategy fitness. Furthermore, Pine Script generation uses raw string interpolation, creating a direct path for code injection if parameter constraints are bypassed.
**Biggest opportunity:** Because the system is actively scaling its optimization engine for bulk hypothesis testing, introducing cryptographic signatures on validation data and strict bounds checking on external ingestion now will close the most critical attack surface before it is amplified by scale.
**Biggest blocker:** The architecture assumes a steady, benevolent state; without atomic file locks, concurrent evolutionary scripts will inevitably corrupt the shared JSON state files, and without integrity verification on the markers, the entire premise of the "Montauk Engine" can be compromised by a low-privileged process modifying a single CSV.
