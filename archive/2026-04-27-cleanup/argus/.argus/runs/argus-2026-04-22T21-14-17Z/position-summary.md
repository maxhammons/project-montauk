# Position Summary

## Pragmatist — Round 1 Position

**Claim:** Montauk will not become a trustworthy TECL decision factory until it deletes the extra ceremony around validation and leaderboard state and collapses back to one honest admission path, because right now the code spends too much effort simulating rigor after parts of that rigor stopped carrying real force.

**Evidence:** The North Star says the project wants a governed pipeline that can stress-test, rank, certify, and then hand a human a clean `risk_on` / `risk_off` contract. The problem is that the evidence machine is carrying duplicated policy and advisory checks that still cost like hard gates. In `scripts/validation/pipeline.py`, cross-asset was explicitly removed from `composite_confidence`, marker shape now always returns `PASS`, and walk-forward hard fails were demoted into warnings, yet the pipeline still budgets and runs Gate 6 re-optimization across tiers (`_gate6_cross_asset`, `_gate7_synthesis`, and the Gate 6 budget in `run_validation_pipeline`). In `scripts/search/evolve.py`, `scripts/search/grid_search.py`, `scripts/certify/full_sweep.py`, and `scripts/certify/recertify_leaderboard.py`, leaderboard admission logic is restated instead of truly centralized, which is why the repo also needs recertify and full-sweep repair flows around `spike/leaderboard.json`. Then `scripts/search/spike_runner.py` acts as a second orchestrator: it validates, emits artifacts, patches certification after artifact generation, updates the leaderboard, and tries to backfill more artifacts. That is too much plumbing between “candidate exists” and “human can trust the contract.”

**Biggest opportunity:** Delete the duplicate admission and repair layers until there is one canonical path from validated candidate to leaderboard row to artifact bundle. That would make the evidence machine match the North Star instead of narrating around it.

**Biggest blocker:** The current system still treats semantic leftovers as operational necessities. Gate names, certification language, and leaderboard repair scripts all survive after the underlying force of those boundaries weakened, so the team keeps paying complexity cost for trust signals that no longer line up cleanly with control flow.

## Futurist — Round 1 Position

**Claim:** Montauk will miss its North Star of becoming a trustworthy TECL decision factory unless it hardens one canonical meaning for admission, certification, and leaderboard memory now, because the repo is already operationalizing a multi-strategy confidence system while storing contradictory labels as durable truth.

**Evidence:** The North Star says this codebase is trying to become a governed evidence machine with durable leaderboard memory, not just a better backtest. That makes semantic contracts load-bearing. But `docs/validation-philosophy.md` says `backtest_certified` is the correctness flag and `promotion_ready = backtest_certified AND confidence >= 0.70`, while `scripts/validation/pipeline.py::_gate7_synthesis()` sets `promotion_ready = verdict == "PASS"` and only then computes `backtest_certified`. The effect is already visible in `spike/leaderboard.json`: the current board contains 20 rows with `verdict="PASS"` and `promotion_ready=true`, but all 20 have `backtest_certified=false`. At the same time, `docs/project-status.md` still frames the product around a `backtest_certified` signal bundle for the best PASS winner, `scripts/search/evolve.py::_is_leaderboard_eligible()` admits watchlist rows at `composite_confidence >= 0.60`, and `scripts/certify/recertify_leaderboard.py` rewrites leaderboard history under current rules without storing a validation-model version in the row. Trace that forward 18 months: more engineers, more recertifications, more candidate families, and a UI/reporting surface trained on labels that no longer describe the same boundary. That is not documentation drift. That is a future migration project disguised as a naming issue.

**Biggest opportunity:** Define a single persisted trust contract for `verdict`, `promotion_ready`, `backtest_certified`, watchlist admission, and leaderboard eligibility, then stamp that contract with versioned provenance in each leaderboard row. That would give the North Star what it actually needs: durable memory that can survive rule changes without reinterpreting history by archaeology.

**Biggest blocker:** The repo is already writing contradictory semantics into durable state. Once `PASS` rows, recertified leaderboards, and report surfaces accumulate under mixed meanings, tightening the model stops being a refactor and becomes a trust migration across code, docs, and operator behavior.

## Craftsman — Round 1 Position

**Claim:** Montauk cannot become the trustworthy TECL decision factory named in the North Star until it stops describing one validation system and running another, because its core trust boundary is currently a semantic counterfeit: the code has split leaderboard admission from champion certification, but the repo still talks about `gate`, `PASS`, `promotion_ready`, and `backtest_certified` as if they all belong to one clean model.

**Evidence:** The most important thing I discovered is that Gate 7 already implements a two-stage truth the repo has not named honestly. In `scripts/validation/pipeline.py`, `promotion_ready` is set from the PASS verdict while `backtest_certified` is stricter and depends on certification checks that include `artifact_completeness`, which begins life as pending. `scripts/search/evolve.py` then admits leaderboard entries without requiring that artifact-completeness check, and the current `spike/leaderboard.json` proves the point: admitted rows exist with `promotion_ready: true` and `backtest_certified: false`. That is a coherent watchlist-plus-certification model. The problem is that `docs/validation-philosophy.md`, `docs/pipeline.md`, `docs/project-status.md`, and `CLAUDE.md` still export incompatible definitions, while marker and cross-asset are still called gates even after the code demoted them into diagnostics and score inputs. If you read this system cold, you build the wrong mental model first, exactly where the project is trying to manufacture trust.

**Biggest opportunity:** Collapse the vocabulary to match the real architecture: one explicit admission contract for leaderboard memory, one explicit certification contract for the deployable champion, and diagnostic names for checks that no longer veto. That would move Montauk directly toward the North Star because the evidence machine would finally describe its own evidence honestly.

**Biggest blocker:** The blocker is not weak math; it is contradictory authority. As long as `scripts/validation/pipeline.py`, `spike/leaderboard.json`, and the docs teach different meanings for the same words, every future improvement to validation, ranking, or certification will compound confusion instead of trust.

## Accelerator — Round 1 Position

**Claim:** Project Montauk will not become the trustworthy TECL decision factory in the North Star until it collapses its competing trust contracts into one operational boundary, because right now a new engineer can run the system, change the system, and even update the leaderboard without being able to tell which outputs are actually safe to ship.

**Evidence:** The North Star says Montauk is trying to become a governed pipeline that can generate, stress-test, rank, and certify TECL timing strategies, then hand a human operator a clean `risk_on` / `risk_off` contract. That only works if the repo states one unambiguous answer to "what counts as trusted." It does not. `scripts/validation/pipeline.py` defines gate 7 so that only `PASS` entries at `composite_confidence >= 0.70` are `promotion_ready`, while `scripts/search/evolve.py` admits leaderboard entries at `0.60+` so long as four certification checks pass, and `scripts/certify/recertify_leaderboard.py` explicitly rebuilds the board using that looser watchlist rule. The docs are worse: `docs/pipeline.md` says both that the leaderboard is a 0.60+ watchlist and that only PASS / `promotion_ready` entries belong there, while `scripts/README.md` still describes the leaderboard as a binding statement that listed strategies have cleared the full validation pipeline. On top of that, first-run instructions in `CLAUDE.md` and `docs/pipeline.md` still point engineers at commands and files that no longer exist. The important improvement is that `tests/test_regression.py` now gives engineers a real local safety net for engine behavior, which means the code is faster to change than before. But that safety net guards the engine, not the contract. A system that can compute accurately while labeling trust inconsistently is exactly how a "decision factory" turns into a confidence factory.

**Biggest opportunity:** Define one canonical trust surface for `spike/leaderboard.json` and make every doc, admission path, and certification step repeat that same rule verbatim. That is the highest-leverage velocity move because it makes every future search, recertification, and manual execution step cheaper to understand and safer to ship.

**Biggest blocker:** Semantic drift between leaderboard admission, `promotion_ready`, and `backtest_certified`. It blocks the North Star because the project is trying to become an evidence machine, and an evidence machine that cannot state what its own evidence means forces humans to carry the real contract in their heads.

## Exploiter — Round 1 Position

**Claim:** Montauk cannot become the trustworthy TECL decision factory described in the North Star until it stops treating `spike/leaderboard.json` as both a memory surface and a trust boundary, because the current admission and certification chain already allows semantic authority to be laundered across that boundary.

**Evidence:** The North Star says this project is trying to hand a human operator a clean `risk_on` / `risk_off` contract backed by governed evidence, durable leaderboard memory, and certification artifacts. But the code does not hold one clean line between memory and authority. `scripts/README.md` says a leaderboard row means full validation PASS plus certification truth, while `scripts/search/evolve.py` admits rows at `composite_confidence >= 0.60` and even grandfathered legacy entries. That softer rule is repeated in `scripts/search/focus_spike.py`, `scripts/certify/full_sweep.py`, and `scripts/certify/recertify_leaderboard.py`, which means this is policy, not drift at one edge. Then `scripts/search/spike_runner.py` finalizes certification by recomputing `backtest_certified` from the check bundle alone, severing it from Gate 7’s stricter `promotion_ready and all(checks)` rule in `scripts/validation/pipeline.py`. From there, `scripts/certify/certify_champion.py` and `scripts/certify/backfill_artifacts.py` can turn leaderboard rows into fresh run artifacts, and `viz/build_viz.py` plus `viz/templates/app.js` surface those rows to the operator as ranked candidates. That is the attack surface: not shell access, but a confused trust model around persisted state that can make a softer or stale row look authoritative downstream.

**Biggest opportunity:** Collapse leaderboard admission, certification, and artifact generation onto one enforced trust contract. If the repo makes `spike/leaderboard.json` either a strict PASS-and-certified boundary or a clearly separate watchlist/memory surface, the project gets closer to its North Star immediately because every downstream manual decision surface inherits a truthful contract instead of a negotiable one.

**Biggest blocker:** The vocabulary and the implementation no longer agree about what authority means. As long as `PASS`, `promotion_ready`, `backtest_certified`, `watchlist`, and leaderboard membership point at different thresholds in different files, the system cannot honestly act like an evidence machine, because I can still reach the operator through semantic gaps in state handling.
