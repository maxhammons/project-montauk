# Montauk 3.0 Research — Stream 7: Rust sandbox and software supply chain

## 1. One-page plain-English conclusion

Montauk 3.0 already made the right first call: the **normal** authoring path is a typed, declarative `StrategySpec` interpreted by protected Rust, not arbitrary code (`rust-strategy-and-evaluation-policy.md`; D26/D27). A declarative spec has no `Cargo.toml`, no `build.rs`, no proc macros, no dependency graph, nothing to compile — so the **compile-time / build-supply-chain** slice of this stream's threat surface (malicious crates, poisoned build scripts, compiler/linker abuse) does not exist on the normal path. That is narrower than "most of the threat surface is gone," and should be stated that way: the declarative path still *deserializes* an agent-submitted spec and *expands* a constraint graph against it, and both steps are their own threat-modeling subject, not a footnote. Deserializing an untrusted, arbitrarily-shaped document is a documented Rust vulnerability class requiring no compiler, no crate, and no `unsafe` — recursion-depth stack overflows in `serde`-family deserializers (RUSTSEC-2024-0012, -2019-0025, -2022-0004) and the older config-language "expansion bomb" class (CWE-776, "billion laughs") both apply. The `StrategySpec` pipeline's own steps 1 and 5 (deserialize against schema; expand satisfying parameter combinations in bounded batches) are exactly where a deeply-nested logic graph, a combinatorial constraint blow-up, or a batch-size integer overflow could turn a normal spec submission into a denial-of-service against the protected engine, with no escape hatch involved — this needs its own parser-hardening acceptance test (Experiment 7.5), separate from escape-hatch containment. The residual danger from **compiled, executable** code is still concentrated exactly where the charter already flags it as staged-and-disabled: the **isolated Rust family-module escape hatch** (D27, D31, D64) — the only place compilers, crates, `unsafe`, and FFI enter the picture.

**What Max should do:** treat the escape hatch's containment suite as a second, independent, prerequisite certification gate — never mixed into Gold's economic verdict. Concretely: (a) compile every escape-hatch module with **zero third-party crates** — only `std` and the frozen Montauk Strategy SDK — so there is no dependency graph, `build.rs`, or proc macro to poison; (b) execute the compiled binary only inside a disposable worker with OS-level sandboxing (seccomp-bpf + Linux namespaces, no network, no credentials, read-only core mount); (c) require every protected-core release build to be independently reproducible bit-for-bit, so a compromised build environment cannot silently swap in a backdoored core; (d) sign every release and every containment-suite pass with Max's offline key, with a tested revocation/rollback runbook; (e) bound the declarative `StrategySpec` deserializer itself — recursion-depth cap, logic-graph node-count cap, checked non-overflowing expansion-count bound — so a malformed spec cannot stack-overflow or combinatorially stall the engine on the normal path. All five are properties of the *method* (crate policy, sandbox layer, build reproducibility, input bounding) — none is tuned to any specific strategy family or contingent on which strategy currently leads the leaderboard.

**Concrete example:** the agent proposes a mechanism the primitive library cannot express (say, a wavelet-decomposition trend filter). Its Rust source compiles with no network access, using only `std` + the frozen SDK. The binary runs in a seccomp-restricted, network- and credential-denied worker against a replayed historical prefix. If it compiles, stays within its CPU/memory/time budget, and produces identical output across repeated seeded runs, it enters the same untrusted intake queue as a declarative family and still faces every Gold plank. Any containment failure is logged as an *implementation* verdict (quarantine/repair), never an *economic* verdict on the idea — existing 3.0 doctrine this stream should reinforce, not weaken.

The supply-chain half of this stream (SBOM, signing, pinning, reproducible builds) matters almost entirely for the **protected core's own** dependency chain, not for candidates — because a zero-dependency escape hatch has no supply chain of its own. Max's real exposure is (1) a compromised or typosquatted crate entering the core's `Cargo.lock` during a legitimate maintenance release, and (2) a compromised release-signing workflow. Both are solved with ordinary, well-documented practice (`cargo audit`, `cargo vet`, pinned `Cargo.lock`, reproducible builds, cosign signing) applied to the core repo on Max's own maintenance cadence — not a continuous, agent-driven process.

## 3. Recommended Montauk experiments

Scoped strictly to what Stream 7 owns — escape-hatch containment and core supply-chain integrity — not strategy economics. None of these endorse or reject any current strategy.

**7.1 — Escape-hatch containment acceptance suite** (the gate D27/D31 require before the hatch is enabled). *Estimand:* for a fixed, versioned battery of adversarial + benign Rust modules, the pass/fail outcome of: (a) zero network access, (b) zero credential/secret access, (c) zero writes outside the output path, (d) enforced CPU/wall-clock/memory/output limits, (e) bit-identical output across ≥20 seeded reruns, (f) zero reads past the causal prefix (via a planted "future-leak canary"). *Stopping rule:* the battery, limits, and rubric are frozen and hashed before the first run; each module either passes all six checks or fails — no partial credit, no re-running until it passes. A worker-configuration fix requires rerunning the entire suite from scratch, discarding the prior result. **Honest limit on (e):** 20 is a practical floor with no stated reproduction-probability model behind it — a low-probability race could pass 20/20 and still exist in production. Backstopped by treating any *observed* nondeterminism, ever, as a permanent fail (never a retry), and by an SDK-level requirement that escape-hatch modules have no shared mutable state to race on in the first place.

**7.5 — Declarative `StrategySpec` parser-hardening test** (normal path, not the escape hatch). *Estimand:* pass/fail of the deserializer and constraint-expansion step against a fixed battery: (a) a logic graph nested past a declared depth cap, (b) a constraint set whose satisfying combinations exceed a declared expansion-count cap, (c) a batch-size computation pushed toward integer overflow, (d) a schema-valid spec at the boundary of the caps. *Stopping rule:* caps frozen and hashed before the first run; each case must fail closed (structured error, no panic, no unbounded resource growth) — anything else is a fail, not a partial pass. Must pass before any cap change ships, independent of the escape-hatch schedule.

**7.2 — Zero-third-party-crate stress test.** *Estimand:* the fraction of a predefined list of N ≥ 10 candidate "novel mechanism" ideas (drawn from existing backlog, chosen before testing) that compile and run correctly using only `std` + the frozen SDK. *Stopping rule:* the mechanism list and success criterion (compiles, runs, produces a non-degenerate signal on the smoke fixture) are frozen before implementation; report the exact fraction, no substituting easier mechanisms after failures. **Honest limit on N:** at N=10, binomial sampling error is wide (7/10 vs. 9/10 is barely distinguishable) — report the fraction as a coarse, directional signal only. A result at either extreme (≤3/10 or ≥9/10) is informative; a middle result (4–7/10) is not decision-grade and should trigger expanding N, not a policy change on N=10 alone. *Decision consequence:* a low pass rate is evidence for a small, hand-vetted allowlist over open dependency resolution — a graduated response Max decides, not the agent.

**7.3 — Reproducible-build verification of the core.** *Estimand:* whether two independent clean-room builds of the same signed release (same commit, `Cargo.lock`, toolchain) on two different machines produce byte-identical binaries after normalizing only documented variance (`SOURCE_DATE_EPOCH`, fixed build path). *Stopping rule:* exactly two independent builds per release; match certifies reproducibility for that release, mismatch blocks signing until root-caused — no "close enough."

**7.4 — Dependency-audit gate on every core release.** *Estimand:* pass/fail of (a) `cargo audit` returning zero unresolved advisories on the pinned lockfile, and (b) a manual attestation that every new/bumped dependency since the last release was actually read by Max or a delegate. *Stopping rule:* binary, blocks signing, checked once per release candidate against the exact lockfile that will ship.

## 4. Null, defect, and planted-signal controls

- **Positive control:** a benign `std`+SDK-only module (e.g., a naive EMA) must pass Experiment 7.1 cleanly. Failure means the harness is broken, not the module — re-verify after any worker-configuration change.
- **Seeded negatives (must fail for the stated reason, not merely fail):** network-exfiltration attempt (denied by sandbox, not timeout); env/file read outside the allowed path (denied by capability/namespace enforcement); subprocess spawn (denied); one-bar-past-boundary read via the future-leak canary (caught by the causal check, not a silently-too-good backtest); unbounded CPU/memory use (killed by the declared limiter, not the OS OOM-killer as an uncontrolled crash); a seeded data race (caught by the determinism check across reruns).
- **Detecting silent harness cheating:** periodically inject a known-bad module disguised as ordinary research output and confirm it is classified as a containment failure, not routed into ordinary quarantine-and-retry as if it were merely an economic rejection — the specific failure mode the charter already worries about (a resource failure "mislabeled as economic rejection").
- **Supply-chain planted-signal control:** on Experiment 7.4's first run, pin one dependency at a version carrying a known, already-patched RustSec advisory and confirm `cargo audit` actually flags it before relying on the gate for real releases.
- **Declarative-path planted-signal control (7.5):** submit one spec engineered to nest exactly one level past the depth cap and one engineered to expand exactly one past the combination cap; confirm both are rejected with the expected structured error rather than silently accepted, truncated, or crashing the worker — proving the caps are actually wired in, not merely documented.

## 5. False-Gold and false-rejection consequences

- **False "containment passed"** (a leaky escape-hatch module reaches intake believing it is contained) is the worst outcome in this stream: downstream validation assumes causal, side-effect-free evaluation, and a violation could manufacture a **false Gold** for the wrong reason (a data leak, not a real edge) — exactly the failure the whole 3.0 architecture exists to prevent. Bias the acceptance suite hard toward rejection on any ambiguous or flaky result.
- **False "containment failed"** (a safe module blocked by an overly strict sandbox) is bounded and recoverable: the idea sits in repair/quarantine and can be revisited once the harness bug is fixed — a throughput cost, not a trust cost.
- **False "core release is intact"** (a compromised build or dependency reaches a signed release undetected) is on par with the worst outcome above, because every downstream Gold verdict and trusted signal inherits from an untrustworthy core.
- **The asymmetry to keep in mind:** unlike a strategy false-Gold, which is bounded by notional size and correctable through forward monitoring/recertification, a containment or supply-chain false-pass is a **correlated** failure that can undermine every leaderboard row's evidentiary basis at once. Stream 7's gates should be stricter and less negotiable than most individual strategy thresholds, even though they are the least exciting part of the system.
- **A `StrategySpec` parser DoS (7.5's subject) is a different consequence class:** an unbounded-recursion or expansion failure on the normal path is an **availability** failure (engine stalls/crashes on a malformed spec), not an integrity failure — it does not manufacture a false Gold. It still deserves a hard gate, since it could block every family submission rather than one module, but its remediation lane is "fix the cap, redeploy," not a containment or signing incident response.

## 6. Assumptions and power/limits

- **No statistical "sample size" applies to the pass/fail acceptance tests themselves.** Sandbox/containment integrity is proven by adversarial testing, not a frequency estimated from repeated trials — 7.1, 7.3, 7.4, and 7.5 are binary gates and should not carry confidence intervals they cannot honestly support. Two numeric knobs inside those gates are genuinely under-justified rather than derived, and are flagged as such where they appear rather than quietly relied on elsewhere: **[Inference, not derived]** 7.1's "≥20 reruns" and 7.2's "N ≥ 10 mechanisms" (see Section 3's honest-limit notes for each) — neither should be cited in this project as a precision instrument.
- **The zero-third-party-crate policy is a design choice under uncertainty [Inference], not a proven-necessary rule.** Motivated by RUSTSEC-2026-0155 and the structural `build.rs`/proc-macro problem, but Experiment 7.2 has not yet run to confirm it is not too restrictive. **[Inference, not sourced]:** Section 1's claim that compile-time threat surface "does not exist on the normal path" covers only the build/compile slice — it does not extend to parser/deserialization-level DoS against spec ingestion, which is real and is now covered by Experiment 7.5. Likewise, RUSTSEC-2026-0155 is solid primary evidence of a zero-dependency, download-and-execute crate reaching crates.io — it is **not** evidence the incident "defeated" any specific detection heuristic; that framing was this document's own earlier gloss, corrected in Section 2.
- **SLSA's design population does not match Montauk's** [Evidence: primary, inference on transfer]. SLSA targets multi-contributor supply chains with a build-farm operator as a distinct trust boundary; Montauk has one signer and no build farm. Importing L3's multi-party isolation machinery wholesale would add complexity against a threat that does not exist here.
- **Firecracker's density/throughput numbers do not transfer** [Evidence: primary, explicit non-transfer]. They were measured for many concurrent, mutually-untrusted cloud tenants at extreme packing density; Montauk's relevant property is the isolation *model* (hardware-backed VM boundary, minimal device surface), not the packing figures.
- **Wasm/WASI is a live alternative, not a settled recommendation.** Its capability model is structurally strong, but compiling escape-hatch Rust to Wasm is an architecture change beyond any current 3.0 commitment, and Wasmtime's own docs state resource-quota and side-channel protection still require host-layered controls — worth a bounded prototype, not an immediate switch.
- **Low usage frequency argues for depth over automation.** The escape hatch is explicitly an exception path; its low expected volume means investing more rigor per event (not building heavy automated re-testing tooling for a high-volume case that will not materialize) is the right tradeoff.
- **Where inference is not warranted:** no amount of containment testing proves a strategy's economic validity. Stream 7 can only certify that a module behaved as a well-contained, deterministic, causal-only program — this must never substitute for the validation pipeline's economic and anti-overfit planks ("compilation is not validation").

## 7. Required fixtures and durable artifacts

- **Deterministic positive fixture:** one benign `std`+SDK-only module (naive EMA) with a frozen expected output on a frozen historical prefix, retained with its expected result ("pass, all six checks"); re-verified after any worker/sandbox configuration change.
- **Seeded-negative battery:** the six adversarial modules from Section 4, each with a frozen expected *failure reason*, versioned so a fix for one class cannot silently regress another.
- **Future-leak canary fixture:** a synthetic series with a detectable, out-of-band marker planted just past the causal cutoff, so any read past the boundary produces a mechanically-detectable signature rather than a merely too-good backtest.
- **Reproducible-build artifact:** for every signed release, retain both independent build hashes, the toolchain/environment manifest, and the diff (empty if reproducible), alongside the signed manifest.
- **Dependency-audit artifact:** for every signed release, retain `cargo audit` output, the full `Cargo.lock`, a generated SBOM (CycloneDX recommended), and a note naming who reviewed each new/bumped dependency and when.
- **Signing/revocation runbook:** a written, drilled procedure for key rotation, revoking trust in a compromised prior release, and rolling back to last-known-good without error under time pressure — turning the charter's "credential revocation on exit" principle into a tested procedure for the specific case of a bad release.
- **Spec-parser adversarial battery (7.5):** the four `StrategySpec` cases from Experiment 7.5, each with a frozen expected outcome ("rejected, structured error, no unbounded resource growth"), versioned alongside the frozen depth/count caps so a cap change cannot ship without rerunning the battery.

## 8. Unresolved owner decisions

- **Ban all third-party crates in the escape hatch, or allow a small hand-vetted allowlist?** *Default:* ban entirely until Experiment 7.2 shows this is too restrictive. *Tradeoff:* simplest, smallest surface, consistent with the RUSTSEC-2026-0155 lesson that dependency-count heuristics are unreliable — but may occasionally block a mechanism a well-audited crate would make trivial.
- **OS sandboxing alone, or add a Firecracker microVM layer?** *Default:* OS-level sandboxing (seccomp + namespaces + cgroups) first, given low escape-hatch volume; add a microVM only if a documented near-miss or an explicit risk decision justifies the added operational complexity. *Tradeoff:* simplicity now vs. residual kernel-escalation risk accepted.
- **Hardware-backed offline signing key, or passphrase-protected software key on an air-gapped device?** *Default:* hardware-backed key — materially reduces "key exfiltrated from a compromised machine" risk. *Tradeoff:* adds a physical dependency and a pre-planned lost/damaged-device recovery procedure.
- **Full reproducible-build + audit ceremony on every release, or only releases that touch dependencies?** *Default:* every release, regardless of what changed — the marginal cost is small, and "this release didn't touch dependencies so we skipped the check" is exactly the shortcut that lets an incident slip through. *Tradeoff:* small fixed cost added to every maintenance session.
- **Retain SBOMs only locally, or as a durable off-machine artifact per the charter's GitHub recovery-path principle?** *Default:* commit the SBOM alongside the signed manifest in the existing off-machine recovery path. *Tradeoff:* negligible cost; no real downside for a single-owner appliance.
- **What are the actual numeric caps (recursion depth, expansion count) for the `StrategySpec` parser (7.5)?** This document identifies the threat class but not the numbers — they depend on schema shape and legitimate family complexity, outside this stream's scope. *Default:* set caps generously above every current backlog family's observed depth/count, tighten only if 7.5 shows headroom; never leave uncapped. *Tradeoff:* too low blocks legitimate families; too high narrows but doesn't eliminate the DoS window — either way, "no cap" is not acceptable.
